1. Introduction

1.1 Rappel du contexte DevSecOps

Le mouvement DevSecOps prolonge la culture DevOps en intégrant la sécurité (Sec) comme une responsabilité partagée tout au long du cycle de vie logiciel. Plutôt que de confier la sécurité à une équipe isolée intervenant en fin de projet, chaque étape — du commit initial au déploiement en production — intègre des contrôles et vérifications de sécurité automatisés.

En tant que futurs ingénieurs en cybersécurité, vous devez maîtriser non seulement les techniques d'attaque et de défense, mais aussi les processus industriels qui permettent de construire des logiciels sécurisés à grande échelle. C'est exactement ce que ce module vous propose.

3. Présentation globale du scénario

3.1 Description du projet

Vous allez travailler sur SecNotes, une application web de gestion de notes internes d'entreprise. Cette application est volontairement vulnérable : elle simule ce qu'un développeur junior pourrait produire sans formation en sécurité. Votre mission est de l'intégrer dans un pipeline DevSecOps complet pour détecter, documenter et (partiellement) corriger ses failles.

3.2 Pourquoi une application vulnérable ?

Dans un contexte pédagogique, travailler avec du code vulnérable permet de :

  • Observer concrètement ce que les outils SAST détectent (et ce qu'ils manquent)
  • Comprendre les failles en les voyant dans du vrai code, pas seulement en théorie
  • Mesurer l'apport d'un pipeline DevSecOps par rapport à un développement sans contrôle
  • Pratiquer la remédiation guidée sur des vulnérabilités concrètes

3.3 Pourquoi chaque technologie ?

🟢 MongoDB

Base NoSQL très utilisée avec Node.js. Permet de démontrer les injections NoSQL, un type de vulnérabilité moins connu que l'injection SQL classique mais tout aussi dangereux.

🔵 GitLab CI/CD

Plateforme CI/CD intégrée au gestionnaire de code. Le fichier .gitlab-ci.yml permet de décrire tout le pipeline de manière déclarative, versionnée avec le code.

🟡 GitLab Runner local

GitLab SaaS ne peut pas exécuter vos jobs directement. Le Runner, installé sur votre PC, écoute GitLab et exécute réellement les commandes. C'est le « bras » de GitLab sur votre machine.

🟣 SonarQube local

Outil d'analyse statique de code (SAST) qui détecte bugs, code smells et vulnérabilités. Installé localement pour éviter les coûts cloud et garder le contrôle sur l'analyse.

🐳 Docker

Conteneurisation de l'application pour garantir la portabilité et la reproductibilité du déploiement, localement comme en cloud.

☸️ Kubernetes (local → AWS)

Orchestrateur de conteneurs. On commence en local pour apprendre, puis on déploie sur AWS EKS pour simuler un environnement de production.

🏗️ Terraform

Infrastructure as Code : décrire l'infrastructure AWS (réseau, cluster, permissions) sous forme de fichiers versionnés et reproductibles.

⚠️
Attention — Distinction cruciale

Ne confondez pas ces outils : GitLab stocke votre code et définit le pipeline. GitLab Runner exécute les jobs du pipeline sur votre machine. SonarQube analyse votre code. Docker empaquète votre application. Kubernetes orchestre les conteneurs. Terraform crée l'infrastructure cloud. Chacun a un rôle distinct et complémentaire.

🚫
Important — Limites de SonarQube

SonarQube est un excellent outil SAST, mais il ne détecte pas tout. Les vulnérabilités logiques (IDOR, V4), les mauvaises pratiques Docker (V7), les secrets Kubernetes (V9) et les injections NoSQL complexes (V1) échappent souvent à l'analyse statique pure. C'est pourquoi un pipeline DevSecOps complet doit combiner plusieurs outils (SAST + SCA + DAST + secrets scanning + container scanning).

5.6 Extrait de code — Exemple d'injection NoSQL

Pour illustrer la vulnérabilité V1, voici le code vulnérable dans le contrôleur d'authentification :

// authController.js — VERSION VULNÉRABLE
exports.login = async (req, res) => {
  try {
    // ⚠️ DANGER : req.body est passé directement à la requête MongoDB
    // Un attaquant peut envoyer { "username": {"$gt": ""}, "password": {"$gt": ""} }
    // pour contourner l'authentification
    const user = await User.findOne({
      username: req.body.username,
      password: req.body.password     // ⚠️ Pas de hashage + injection possible
    });

    if (user) {
      const token = jwt.sign({ id: user._id }, 'ma-cle-secrete-en-dur');  // ⚠️ V2
      console.log(`Login successful for ${user.username}, token: ${token}`); // ⚠️ V8
      res.json({ token });
    } else {
      res.status(401).json({ error: 'Invalid credentials' });
    }
  } catch (err) {
    console.error('Login error:', err);  // ⚠️ V8 — stack trace exposée
    res.status(500).json({ error: err.message }); // ⚠️ Message d'erreur interne exposé
  }
};

7. Mise en place de GitLab Runner local

7.1 Pourquoi GitLab Runner est indispensable

C'est l'un des concepts les plus mal compris par les débutants : GitLab lui-même n'exécute pas vos jobs CI/CD. GitLab est une plateforme web qui stocke votre code, définit votre pipeline et affiche les résultats. Mais l'exécution réelle des commandes (npm install, npm test, docker build…) est déléguée à un agent appelé GitLab Runner.

🔗 Relation GitLab SaaS ↔ GitLab Runner local
☁️ GitLab SaaS
gitlab.com

• Stocke le code
• Définit le pipeline
• Affiche les résultats
• Distribue les jobs
⇄
🖥️ GitLab Runner (votre PC)
Processus local

• Interroge GitLab régulièrement
• Récupère les jobs en attente
• Exécute les commandes
• Renvoie les résultats
Communication HTTPS sortante (votre PC → gitlab.com). Pas besoin d'ouvrir de ports entrants.

7.6 Erreurs fréquentes avec GitLab Runner

❌
Erreur : « This job is stuck because you don't have any active runners »

Cause : Aucun Runner enregistré ou Runner hors ligne.
Solution : Vérifiez avec gitlab-runner list et gitlab-runner verify. Redémarrez avec gitlab-runner restart.

❌
Erreur : « Job failed: exit status 1 » sur npm ou docker

Cause : L'outil n'est pas trouvé dans le PATH de l'utilisateur qui exécute le Runner.
Solution (Windows) : Assurez-vous que le service GitLab Runner tourne avec un utilisateur qui a les outils dans son PATH. Relancez le service après modification du PATH.
Solution (Linux) : Le runner s'exécute souvent en tant qu'utilisateur gitlab-runner. Ajoutez cet utilisateur aux groupes nécessaires : sudo usermod -aG docker gitlab-runner.

❌
Erreur : Le runner apparaît « offline » dans GitLab

Cause : Le processus Runner ne tourne plus, ou un firewall bloque la connexion sortante vers gitlab.com.
Solution : gitlab-runner run (mode premier plan pour déboguer) ou gitlab-runner restart. Vérifiez votre connexion réseau et proxy éventuel.

8.7 Limites de SonarQube

⚠️
Ce que SonarQube ne détectera PAS dans SecNotes
  • Injections NoSQL complexes — SonarQube analyse le code statiquement et ne comprend pas toujours le flux de données vers MongoDB
  • IDOR (V4) — C'est une vulnérabilité de logique métier, invisible pour un analyseur statique

  • Dépendances vulnérables — SonarQube fait du SAST, pas du SCA (utilisez npm audit ou des outils dédiés)

Voici le pipeline complet, commenté bloc par bloc. Chaque section est expliquée en détail.

# ==============================================================
# Pipeline DevSecOps complet — SecNotes
# ==============================================================

# Définition des stages dans l'ordre d'exécution
stages:
  - prepare
  - test
  - sast
  - build
  - deploy_local
  - infra_aws
  - deploy_aws

# Variables globales
variables:
  APP_NAME: "secnotes"
  DOCKER_IMAGE: "secnotes-app"
  DOCKER_TAG: "$CI_COMMIT_SHORT_SHA"
  K8S_NAMESPACE: "secnotes"

# ---- STAGE 1 : PRÉPARATION ----
# Ce job installe les dépendances npm.
# Le dossier node_modules est mis en cache pour accélérer les builds suivants
# et en artifact pour être disponible dans les jobs suivants.
install_deps:
  stage: prepare
  tags:
    - local          # S'exécute sur notre Runner local taggé "local"
  script:
    - echo "📦 Installation des dépendances..."
    - cd secnotes-vuln-app
    - npm ci          # npm ci est préféré à npm install en CI (plus rapide, déterministe)
  artifacts:
    paths:
      - secnotes-vuln-app/node_modules/
    expire_in: 1 hour
  cache:
    key: "$CI_COMMIT_REF_SLUG"
    paths:
      - secnotes-vuln-app/node_modules/

# ---- STAGE 2 : TESTS UNITAIRES ----
# Exécute les tests Jest. Si un test échoue, le pipeline s'arrête.
# C'est le premier garde-fou : pas de code cassé en production.
unit_tests:
  stage: test
  tags:
    - local
  dependencies:
    - install_deps    # Récupère les artifacts (node_modules) du job précédent
  script:
    - echo "🧪 Exécution des tests unitaires..."
    - cd secnotes-vuln-app
    - npm test -- --coverage --forceExit
  artifacts:
    paths:
      - secnotes-vuln-app/coverage/
    expire_in: 1 week
    when: always       # Conserve le rapport même si les tests échouent

# ---- STAGE 3 : ANALYSE SAST SONARQUBE ----
# Envoie le code à SonarQube local pour analyse statique.
# Le Quality Gate détermine si le code est assez « propre » pour continuer.
sonarqube_scan:
  stage: sast
  tags:
    - local
  dependencies:
    - install_deps
    - unit_tests       # Pour récupérer le rapport de couverture
  script:
    - echo "🔍 Analyse SAST avec SonarQube..."
    - cd secnotes-vuln-app
    - >
      sonar-scanner
      -Dsonar.host.url=$SONAR_HOST_URL
      -Dsonar.token=$SONAR_TOKEN
      -Dsonar.projectKey=secnotes
      -Dsonar.sources=src
      -Dsonar.tests=tests
      -Dsonar.javascript.lcov.reportPaths=coverage/lcov.info

# ---- STAGE 4 : BUILD DOCKER ----
# Construit l'image Docker de l'application.
# L'image est taguée avec le SHA court du commit pour la traçabilité.
docker_build:
  stage: build
  tags:
    - local
  script:
    - echo "🐳 Construction de l'image Docker..."
    - cd secnotes-vuln-app
    - docker build -t $DOCKER_IMAGE:$DOCKER_TAG .
    - docker tag $DOCKER_IMAGE:$DOCKER_TAG $DOCKER_IMAGE:latest

# Applique les manifests Kubernetes pour déployer sur le cluster local.
k8s_local_deploy:
  stage: deploy_local
  tags:
    - local
  dependencies:
    - docker_build

9.4 Explication pédagogique des concepts clés

Le mot-clé dependencies

Dans GitLab CI, dependencies permet de spécifier quels jobs précédents fournissent des artifacts au job courant. Cela évite de télécharger tous les artifacts de tous les jobs précédents et garantit que le job a accès aux fichiers dont il a besoin.

Le mot-clé artifacts

Les artifacts sont des fichiers produits par un job et conservés par GitLab. Ils peuvent être :

  • Transmis aux jobs suivants (ex: node_modules, rapports de couverture)
  • Téléchargés depuis l'interface GitLab pour inspection
  • Expirés automatiquement après un délai configurable

Le mot-clé when: manual

Un job marqué when: manual ne s'exécute que lorsqu'un utilisateur clique sur « Play » dans l'interface GitLab. C'est essentiel pour les jobs coûteux (Terraform sur AWS) ou irréversibles (déploiement en production).

Le mot-clé allow_failure: true

Permet au pipeline de continuer même si ce job échoue. Utilisé ici pour SonarQube en phase d'apprentissage. En production réelle, il faudrait retirer cette option et configurer un Quality Gate strict.

Le mot-clé tags

Les tags permettent de diriger un job vers un Runner spécifique. Ici, tags: [local] assure que le job est pris en charge par votre Runner local (celui enregistré avec le tag « local » à la section 7).

🎓
Point pédagogique

La beauté d'un pipeline CI/CD bien conçu est qu'il fonctionne comme un filet de sécurité automatisé. Chaque git push déclenche automatiquement toute la chaîne de vérification. Si un test échoue ou si SonarQube détecte une vulnérabilité critique, le pipeline s'arrête avant que le code défaillant n'atteigne la production.

  • Le pipeline comporte 7 stages ordonnés logiquement
  • Les variables CI/CD protègent les secrets (tokens, clés AWS)
  • Les artifacts transmettent les fichiers entre jobs
  • Les stages AWS sont en déclenchement manuel pour éviter les coûts accidentels
  • Le tag local dirige tous les jobs vers votre Runner local

11. Automatisation AWS avec Terraform

11.1 Pourquoi Terraform ?

Infrastructure as Code (IaC) signifie décrire votre infrastructure cloud sous forme de fichiers texte versionnés, plutôt que de cliquer dans une console web. Terraform est l'outil IaC le plus populaire et le plus polyvalent.

ApprocheCréation manuelle (console AWS)Infrastructure as Code (Terraform)
Reproductibilité❌ Dépend de la mémoire humaine✅ Identique à chaque exécution
Versionnement❌ Pas d'historique✅ Versionné avec Git
Revue❌ Pas de revue possible✅ Code review comme pour le code applicatif
Automatisation❌ Manuelle✅ Intégrable dans un pipeline CI/CD
Documentation❌ Externe nécessaire✅ Le code EST la documentation
Destruction propre⚠️ Risque d'oubli de ressources✅ terraform destroy supprime tout

11.2 Ce que Terraform va créer

Notre configuration Terraform va provisionner l'infrastructure suivante sur AWS :

VPC + Subnets

Réseau virtuel isolé avec 2 subnets publics dans 2 zones de disponibilité pour la haute disponibilité.

Internet Gateway

Passerelle permettant aux ressources du VPC d'accéder à Internet et d'être accessibles depuis l'extérieur.

EKS Cluster

Cluster Kubernetes managé par AWS. Le control plane est géré par AWS, vous ne gérez que les nœuds de travail.

Node Group

Groupe de nœuds EC2 (2× t3.medium) qui exécutent vos pods Kubernetes.

IAM Roles

Rôles et politiques IAM pour que EKS et les nœuds puissent accéder aux services AWS nécessaires.

Security Groups

Règles de pare-feu contrôlant le trafic réseau entrant et sortant du cluster.

11.5 État Terraform (tfstate)

Terraform maintient un fichier d'état (terraform.tfstate) qui fait le lien entre votre code et les ressources réelles sur AWS. Ce fichier est critique :

  • Ne le supprimez pas : Terraform ne saurait plus quelles ressources il gère
  • Ne le commitez pas dans Git (il peut contenir des secrets)
  • En production, stockez-le dans un backend distant (S3 + DynamoDB pour le verrouillage)
❌
Erreur fréquente : « Error: No valid credential sources found »

Cause : Terraform ne trouve pas les credentials AWS.
Solution : Vérifiez que aws configure a été exécuté ou que les variables d'environnement AWS_ACCESS_KEY_ID et AWS_SECRET_ACCESS_KEY sont définies.

❌
Erreur fréquente : « Error: creating EKS Cluster: AccessDeniedException »

Cause : Votre utilisateur IAM n'a pas les permissions suffisantes.
Solution : L'utilisateur IAM doit avoir les politiques AmazonEKSClusterPolicy, AmazonVPCFullAccess, IAMFullAccess (ou AdministratorAccess pour le TP, à restreindre en production).

13.3 Remédiations recommandées

V1 — NoSQL Injection → Validation et sanitization

// AVANT (vulnérable)
const user = await User.findOne({ username: req.body.username });

// APRÈS (sécurisé) — Utiliser mongo-sanitize
const sanitize = require('mongo-sanitize');
const user = await User.findOne({ username: sanitize(req.body.username) });

V2 — Secrets en dur → Variables d'environnement

// AVANT (vulnérable)
const JWT_SECRET = 'ma-cle-secrete-en-dur';

// APRÈS (sécurisé) — Lire depuis l'environnement
const JWT_SECRET = process.env.JWT_SECRET;
if (!JWT_SECRET) throw new Error('JWT_SECRET environment variable is required');

V6 — Hashage faible → bcrypt

// AVANT (vulnérable)
const hashedPassword = md5(password);

// APRÈS (sécurisé)
const bcrypt = require('bcrypt');
const hashedPassword = await bcrypt.hash(password, 12);

V7 — Dockerfile non sécurisé → Multi-stage + utilisateur non-root

# AVANT (vulnérable)
FROM node:18
COPY . .
RUN npm install
CMD ["node", "src/app.js"]

# APRÈS (sécurisé) — Multi-stage, non-root, .dockerignore
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production

FROM node:18-alpine
RUN addgroup -g 1001 appgroup && adduser -u 1001 -G appgroup -D appuser
WORKDIR /app
COPY --from=builder /app/node_modules ./node_modules
COPY src/ ./src/
USER appuser
EXPOSE 3000
CMD ["node", "src/app.js"]

13.4 Limites de l'approche DevSecOps

⚠️
Un pipeline DevSecOps ne remplace pas tout
  • Faux positifs : SonarQube peut signaler du code qui n'est pas réellement vulnérable en contexte. Il faut de l'expertise humaine pour trier.
  • Faux négatifs : Des vulnérabilités logiques (IDOR, business logic flaws) échappent à toute analyse automatisée.
  • Configuration : La sécurité de l'infrastructure (Security Groups, IAM, réseau) doit être auditée séparément.
  • Runtime : Les vulnérabilités exploitables uniquement en runtime nécessitent du DAST ou du pentesting.
  • Supply chain : La sécurité des dépendances nécessite des outils SCA dédiés (npm audit, Snyk, Dependabot).

📋 Récapitulatif — Section 13

  • Le pipeline détecte certaines vulnérabilités mais pas toutes
  • Les remédiations suivent les bonnes pratiques : sanitization, variables d'environnement, bcrypt, Docker multi-stage
  • Un pipeline DevSecOps est nécessaire mais pas suffisant
  • Il doit être complété par des audits manuels, du DAST et de la revue de code

15.2 Difficultés prévisibles

  • Ressources machine : Faire tourner Docker + SonarQube + Kubernetes simultanément demande beaucoup de RAM. Fermez les applications non essentielles.
  • Réseau et proxy : Si vous êtes derrière un proxy d'entreprise ou universitaire, GitLab Runner et Docker peuvent avoir du mal à se connecter. Configurez les variables HTTP_PROXY / HTTPS_PROXY.
  • Permissions : Sur Linux, l'utilisateur gitlab-runner n'a souvent pas accès à Docker. N'oubliez pas usermod -aG docker gitlab-runner.
  • Temps de build : Le premier npm ci et le premier docker build sont lents (téléchargement). Les builds suivants sont plus rapides grâce au cache.
