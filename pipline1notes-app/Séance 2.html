PARTIE 1 : Comprendre le DevSecOps

Objectif de cette partie :

Comprendre ce qu'est le DevSecOps, pourquoi c'est important, et comment cela change la façon dont nous développons et sécurisons les applications.

1.1 - Le monde AVANT DevSecOps

Comment ça fonctionnait avant ?

Imaginez une entreprise qui veut créer une application web. Voici comment ça se passait traditionnellement :

Développeurs

Écrivent le code

Durée : 6 mois

→
Équipe Ops

Déploient l'application

Durée : 2 semaines

→
Équipe Sécurité

Testent la sécurité

Durée : 1 mois

Les problèmes de cette approche :
  • Très lent : 7-8 mois pour mettre en production
  • Bugs découverts tard : Coûteux à corriger
  • Sécurité en dernier : Si on découvre une faille, il faut tout recommencer
  • Mauvaise communication : Les équipes ne se parlent pas assez
  • Coût élevé : Corriger un bug en production coûte 100x plus cher qu'en développement

Exemple concret d'un problème

Scénario : Une entreprise développe un site e-commerce pendant 6 mois.

Les développeurs finissent le code et le donnent à l'équipe Ops.

L'équipe Ops le déploie sur les serveurs.

L'équipe Sécurité fait un audit et découvre une faille SQL Injection critique.

Résultat : Il faut repasser par les développeurs, corriger le code, retester, redéployer... Cela ajoute 2-3 mois de retard et des coûts importants !

1.2 - Qu'est-ce que le DevSecOps ?

Définition simple

DevSecOps est une façon de travailler qui combine :

  • Dev (Development) = Développement
  • Sec (Security) = Sécurité
  • Ops (Operations) = Opérations/Déploiement
L'idée principale :

Au lieu de penser à la sécurité à la fin, on l'intègre dès le début et tout au long du développement !

Le nouveau modèle DevSecOps

Collaboration Continue

Dev + Sec + Ops travaillent ENSEMBLE

Code

+Tests sécu

→
Test

+Scan vulnérabilités

→
Build

+Analyse dépendances

→
Deploy

+Monitoring sécu

Ce cycle se répète automatiquement plusieurs fois par jour !

Les 4 principes clés du DevSecOps :
  1. Shift-Left Security : Penser à la sécurité dès le début
  2. Automatisation : Les tests de sécurité se font automatiquement
  3. Collaboration : Tout le monde est responsable de la sécurité
  4. Rapidité : Déployer vite, mais en toute sécurité

1.3 - Le concept de "Shift-Left Security"

Qu'est-ce que "Shift-Left" ?

"Shift-Left" signifie littéralement "déplacer vers la gauche". Mais vers la gauche de quoi ?

La timeline du développement

←
GAUCHE
Début du projet
→
DROITE
Production
Shift-Left
Sécurité dès le design
Approche classique
Sécurité à la fin

Moment de détection Coût de correction Temps de correction
Pendant le design 1x (Référence) 1 heure
Pendant le développement 5x plus cher 5 heures

Étape 2 : Créer le fichier tasks.js (logique métier)

Ce fichier contient les fonctions pour gérer les tâches : 

// Fichier: tasks.js
// Ce fichier gère la logique de notre application

// Un tableau pour stocker les tâches (en mémoire pour simplifier)
let tasks = [];

// Fonction pour obtenir toutes les tâches
function getAllTasks() {
    return tasks;
}

// Fonction pour ajouter une nouvelle tâche
function addTask(title) {
    // Validation:  le titre ne doit pas être vide
    if (! title || title.trim() === '') {
        throw new Error('Le titre de la tâche ne peut pas être vide');
    }
    
    // Créer un nouvel objet tâche
    const newTask = {
        id: Date.now(), // ID unique basé sur le timestamp
        title:  title.trim(),
        createdAt: new Date()
    };
    
    // Ajouter la tâche au tableau
    tasks.push(newTask);
    
    return newTask;
}

// Fonction pour supprimer une tâche
function deleteTask(id) {
    // Trouver l'index de la tâche
    const index = tasks.findIndex(task => task.id === parseInt(id));
    
    // Si la tâche n'existe pas
    if (index === -1) {
        throw new Error('Tâche non trouvée');
    }
    
    // Supprimer la tâche
    tasks.splice(index, 1);
    
    return true;
}

// Fonction pour réinitialiser les tâches (utile pour les tests)
function resetTasks() {
    tasks = [];
}

// Exporter les fonctions pour les utiliser ailleurs
module.exports = {
    getAllTasks,
    addTask,
    deleteTask,
    resetTasks
};

Étape 3 : Créer le serveur web (app.js)

Ce fichier crée le serveur web et définit les routes (URLs) de l'application :

// Fichier: app.js
// Ce fichier crée le serveur web

const express = require('express');
const { getAllTasks, addTask, deleteTask } = require('./tasks');

// Créer une application Express
const app = express();
const PORT = process.env.PORT || 3000;

// Middleware pour parser le JSON
app.use(express.json());

// Servir les fichiers statiques (HTML, CSS)
app.use(express.static('public'));

// ROUTE 1: Obtenir toutes les tâches
// GET /api/tasks
app.get('/api/tasks', (req, res) => {
    try {
        const tasks = getAllTasks();
        res.json(tasks);
    } catch (error) {
        res.status(500).json({ error: error.message });
    }
});

// ROUTE 2: Ajouter une nouvelle tâche
// POST /api/tasks
app. post('/api/tasks', (req, res) => {
    try {
        const { title } = req.body;
        const newTask = addTask(title);
        res.status(201).json(newTask);
    } catch (error) {
        res.status(400).json({ error: error.message });
    }
});

// ROUTE 3: Supprimer une tâche
// DELETE /api/tasks/: id
app.delete('/api/tasks/:id', (req, res) => {
    try {
        deleteTask(req.params.id);
        res.status(200).json({ message: 'Tâche supprimée' });
    } catch (error) {
        res.status(404).json({ error: error.message });
    }
});

// Démarrer le serveur
if (require.main === module) {
    app.listen(PORT, () => {
        console.log(`Serveur démarré sur le port ${PORT}`);
    });
}

// Exporter l'app pour les tests
module.exports = app;

Étape 5 : Créer les tests (tasks.test.js)

Les tests vérifient que notre code fonctionne correctement : 

// Fichier: tasks.test.js
// Tests automatiques pour notre application

const { getAllTasks, addTask, deleteTask, resetTasks } = require('./tasks');

// Avant chaque test, on réinitialise les tâches
beforeEach(() => {
    resetTasks();
});

// TEST 1: Vérifier qu'on peut ajouter une tâche
test('Doit ajouter une nouvelle tâche', () => {
    const task = addTask('Acheter du lait');
    
    expect(task).toHaveProperty('id');
    expect(task.title).toBe('Acheter du lait');
    expect(getAllTasks()).toHaveLength(1);
});

// TEST 2: Vérifier qu'on ne peut pas ajouter une tâche vide
test('Ne doit pas ajouter une tâche vide', () => {
    expect(() => addTask('')).toThrow();
    expect(() => addTask('   ')).toThrow();
    expect(getAllTasks()).toHaveLength(0);
});

// TEST 3: Vérifier qu'on peut supprimer une tâche
test('Doit supprimer une tâche existante', () => {
    const task = addTask('Faire les courses');
    deleteTask(task.id);
    
    expect(getAllTasks()).toHaveLength(0);
});

// TEST 4: Vérifier qu'on ne peut pas supprimer une tâche inexistante
test('Doit lever une erreur si la tâche n\'existe pas', () => {
    expect(() => deleteTask(999)).toThrow('Tâche non trouvée');
});

// TEST 5: Vérifier qu'on peut obtenir toutes les tâches
test('Doit retourner toutes les tâches', () => {
    addTask('Tâche 1');
    addTask('Tâche 2');
    addTask('Tâche 3');
    
    expect(getAllTasks()).toHaveLength(3);
});

2.3 - Maintenant, qu'est-ce que le CI/CD ?

Le problème sans CI/CD

Actuellement, nous devons :

  • Manuellement lancer les tests : npm test
  • Manuellement vérifier la sécurité
  • Manuellement créer une image Docker
  • Manuellement déployer l'application sur un serveur

Problème : Si on oublie une étape, on peut déployer du code buggé ou non sécurisé !

La solution : CI/CD

CI/CD va automatiser toutes ces étapes. À chaque fois qu'on envoie du code vers GitLab, le système va automatiquement :

1. Build (Construction)

Installer les dépendances et vérifier que le code compile

↓
2. Test

Exécuter automatiquement tous les tests

↓
3. Security (Sécurité)

Scanner le code pour détecter les vulnérabilités et secrets

↓
4. Package

Créer une image Docker de l'application

↓
5. Deploy (Déploiement)

Déployer automatiquement l'application sur le serveur

2.4 - Créer le Dockerfile pour empaqueter l'application

Qu'est-ce qu'un Dockerfile ?

Un Dockerfile est une recette qui dit à Docker comment créer une "image" de notre application. Cette image contient tout ce dont l'application a besoin pour fonctionner.

# Fichier: Dockerfile
# Recette pour créer une image Docker de notre application

# ÉTAPE 1: Partir d'une image Node.js de base
FROM node: 18-alpine

# ÉTAPE 2: Définir le répertoire de travail
WORKDIR /app

# ÉTAPE 3: Copier les fichiers de dépendances
COPY package*.json ./

# ÉTAPE 4: Installer les dépendances
RUN npm install --production

# ÉTAPE 5: Copier le reste du code
COPY .  .

# ÉTAPE 6: Exposer le port 3000
EXPOSE 3000

# ÉTAPE 7: Définir la commande pour démarrer l'application
CMD ["node", "app.js"]

Pour tester localement :

# Construire l'image Docker
docker build -t todo-app: 1.0 .

# Lancer un conteneur
docker run -p 3000:3000 todo-app:1.0

# L'application est maintenant accessible sur http://localhost:3000

2.5 - Créer le pipeline CI/CD avec GitLab

Configuration complète du pipeline

Créer le fichier . gitlab-ci.yml à la racine du projet :

# Fichier: .gitlab-ci. yml
# Configuration du pipeline CI/CD pour notre application

# Définir les étapes du pipeline
stages:
  - build    # Étape 1: Construire
  - test     # Étape 2: Tester
  - security # Étape 3: Vérifier la sécurité
  - package  # Étape 4: Créer l'image Docker
  - deploy   # Étape 5: Déployer

# Variables globales
variables:
  APP_NAME: "todo-app"
  APP_VERSION: "1.0.0"

############################################
# STAGE 1: BUILD
############################################
build-job:
  stage: build
  image: node:18
  script: 
    - echo "Installation des dépendances..."
    - npm install
    - echo "Vérification que le code n'a pas d'erreurs de syntaxe..."
    - node -c app.js
    - node -c tasks.js
    - echo "Build terminé avec succès!"
  artifacts: 
    # Sauvegarder node_modules pour les étapes suivantes
    paths:
      - node_modules/
    expire_in:  1 hour

############################################
# STAGE 2: TEST
############################################
test-job: 
  stage: test
  image: node:18
  script:
    - echo "Exécution des tests unitaires..."
    - npm test
    - echo "Tous les tests sont passés!"
  coverage:  '/All files[^|]*\|[^|]*\s+([\d\.]+)/'

############################################
# STAGE 3: SECURITY
############################################
security-secrets-scan:
  stage: security
  image: zricethezav/gitleaks:latest
  script:
    - echo "Recherche de secrets dans le code..."
    - gitleaks detect --source=.  --verbose --no-git
    - echo "Aucun secret détecté!"
  allow_failure: false

security-dependencies-scan:
  stage: security
  image: node:18
  script:
    - echo "Vérification des vulnérabilités dans les dépendances..."
    - npm audit --audit-level=high
    - echo "Aucune vulnérabilité critique détectée!"
  allow_failure:  false

############################################
# STAGE 4: PACKAGE
############################################
package-docker:
  stage: package
  image: docker:latest
  services:
    - docker:dind
  script:
    - echo "Construction de l'image Docker..."
    - docker build -t $APP_NAME:$APP_VERSION .
    - docker build -t $APP_NAME:latest .
    - echo "Image Docker créée avec succès!"
    - docker images
  only: 
  - main

############################################
# STAGE 5: DEPLOY
############################################
deploy-staging:
stage: deploy
image: docker:latest
services:
  - docker:dind
script:
  - echo "Déploiement en environnement de staging..."
  - echo "Arrêt de l'ancienne version si elle existe..."
  - docker stop todo-app-staging || true
  - docker rm todo-app-staging || true
  - echo "Démarrage de la nouvelle version..."
  - docker run -d --name todo-app-staging -p 3000:3000 $APP_NAME:$APP_VERSION
  - echo "Application déployée sur staging!"
environment:
  name: staging
  url: http://staging.todo-app.local:3000
only: 
  - main

deploy-production:
stage: deploy
image: docker:latest
services: 
  - docker:dind
script:
  - echo "Déploiement en PRODUCTION..."
  - echo "Arrêt de l'ancienne version..."
  - docker stop todo-app-prod || true
  - docker rm todo-app-prod || true
  - echo "Démarrage de la nouvelle version..."
  - docker run -d --name todo-app-prod -p 8080:3000 $APP_NAME:$APP_VERSION
  - echo "Application déployée en PRODUCTION!"
environment:
  name: production
  url: http://www.todo-app.com
when: manual # Nécessite une validation manuelle
only:
  - main

Explication détaillée du pipeline

Stage 1: Build

Objectif : Vérifier que le code peut être construit

Actions :

  • Installe toutes les dépendances avec npm install
  • Vérifie la syntaxe du code JavaScript
  • Sauvegarde node_modules/ pour les étapes suivantes

Si ça échoue : Le code a des erreurs de syntaxe ou des dépendances manquantes

Stage 2: Test

Objectif : Vérifier que le code fonctionne correctement

Actions :

  • Exécute tous les tests avec npm test
  • Vérifie que toutes les fonctions font ce qu'elles doivent faire
  • Calcule la couverture de code (pourcentage de code testé)

Si ça échoue : Au moins un test ne passe pas, il y a un bug dans le code

Stage 3: Security

Objectif : Vérifier qu'il n'y a pas de problèmes de sécurité

Actions :

  • Job 1 : GitLeaks cherche des secrets (mots de passe, clés API) dans le code
  • Job 2 : npm audit vérifie que les bibliothèques utilisées n'ont pas de vulnérabilités connues

Si ça échoue : Il y a un secret en clair ou une bibliothèque vulnérable

Stage 4: Package

Objectif : Créer une image Docker de l'application

Actions :

  • Construit une image Docker contenant l'application et toutes ses dépendances
  • Tag l'image avec la version et "latest"
  • L'image peut maintenant être déployée n'importe où

Si ça échoue : Problème dans le Dockerfile

Stage 5: Deploy

Objectif : Mettre l'application en ligne

Actions :

  • Deploy staging : Déploiement automatique sur l'environnement de test
  • Deploy production : Déploiement manuel sur l'environnement de production (nécessite un clic)

Pourquoi staging : Permet de tester en conditions réelles avant la production

PARTIE 4 : Synthèse et Concepts Clés

Ce que vous avez appris

1. DevSecOps

  • Définition : Approche qui intègre la sécurité dès le début du développement
  • Shift-Left : Penser à la sécurité tôt coûte 100x moins cher
  • Collaboration : Dev, Sec et Ops travaillent ensemble
  • Automatisation : Les contrôles de sécurité sont automatiques

2. CI/CD (Continuous Integration / Continuous Deployment)

  • CI (Continuous Integration) : Intégrer et tester le code plusieurs fois par jour
  • CD (Continuous Deployment) : Déployer automatiquement le code validé
  • Pipeline : Séquence automatisée d'étapes (build → test → security → package → deploy)
  • Avantages : Rapidité, qualité, sécurité, coûts réduits

3. Les étapes d'un pipeline CI/CD

Étape Objectif Outils
Build Compiler et construire l'application npm, maven, gradle
Test Vérifier que le code fonctionne Jest, JUnit, PyTest
Security Détecter les vulnérabilités GitLeaks, npm audit, SonarQube
Package Créer une image déployable Docker
Deploy Mettre en production Kubernetes, Docker

4. Sécurité dans le pipeline

  • Détection de secrets : GitLeaks cherche les mots de passe en clair
  • Scan de dépendances : npm audit vérifie les bibliothèques vulnérables
  • Security Gate : Point de contrôle qui bloque le code non sécurisé
  • Principe : Mieux vaut bloquer en développement qu'en production

Vocabulaire à retenir

Terme Signification
DevSecOps Development + Security + Operations - Approche intégrant la sécurité dans le DevOps
CI/CD Continuous Integration / Continuous Deployment - Automatisation du développement au déploiement
Pipeline Séquence d'étapes automatisées pour traiter le code
Stage Étape majeure du pipeline (build, test, deploy... )
Job Tâche spécifique dans un stage
Artifact Fichier sauvegardé entre les stages du pipeline
Security Gate Point de contrôle qui bloque le code non conforme
Shift-Left Intégrer la sécurité tôt dans le cycle de développement
Secret Information sensible (mot de passe, clé API, token)
Docker Technologie de conteneurisation d'applications
Image Docker Package contenant l'application et ses dépendances
Staging Environnement de pré-production pour tester
Production Environnement accessible aux utilisateurs finaux
Points clés à retenir
  • Le DevSecOps intègre la sécurité dès le début, pas à la fin
  • Le CI/CD automatise tout le processus du code à la production
  • Un pipeline a plusieurs stages qui s'exécutent séquentiellement
  • Si un stage échoue, le pipeline s'arrête (protection)
  • Les tests et la sécurité sont vérifiés automatiquement
  • Corriger un bug tôt coûte 100x moins cher qu'en production
  • GitLab CI/CD utilise le fichier .gitlab-ci.yml pour la configuration
  • Docker permet de packager l'application pour un déploiement uniforme
