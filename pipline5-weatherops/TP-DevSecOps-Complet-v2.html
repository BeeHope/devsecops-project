1. Objectifs pédagogiques et résultats attendus

But du TP — Réaliser une chaîne DevSecOps complète : développement d’une application Node.js, tests automatisés, conteneurisation Docker, pipeline GitLab (qualité et sécurité), publication de l’image dans le GitLab Container Registry, création de l’infrastructure cloud avec Terraform (AWS pour le staging, Azure pour la production), et déploiement de l’image sur ces deux environnements.

Compétences visées — À l’issue du TP, l’étudiant est capable de :

  • Structurer une application Node.js/Express (séparation app / server, variables d’environnement).
  • Écrire des tests automatisés avec Jest et Supertest.
  • Conteneuriser l’application avec un Dockerfile et exploiter le GitLab Container Registry.
  • Définir un pipeline GitLab CI/CD avec les stages test, security, build et déploiement.
  • Décrire l’infrastructure (AWS EC2, Azure App Service) en Terraform et déployer l’image Docker depuis le registry.
  • Expliquer le parcours de l’image : construction dans le pipeline, stockage dans le registry, récupération par AWS et Azure.
Résultats attendus — Application déployée et accessible : route / et route /health fonctionnelles en local, dans un conteneur Docker, sur l’environnement de staging (AWS) et sur l’environnement de production (Azure). Pipeline GitLab vert (tests, sécurité, build, push registry). Infrastructure créée et détruite proprement avec Terraform.

2. Architecture cible de la solution

La solution comporte : un dépôt GitLab contenant le code source et le fichier de pipeline ; un pipeline CI/CD qui exécute les tests, le contrôle de sécurité, le build de l’image Docker et sa publication dans le GitLab Container Registry ; deux environnements cloud créés par Terraform — staging sur AWS (instance EC2) et production sur Azure (App Service for Containers) — qui récupèrent l’image depuis le registry et exécutent le conteneur.

Figure 1 — Architecture DevSecOps globale

flowchart LR subgraph dev[Code] A[Développeur] B[GitLab repo] end subgraph ci[CI/CD] C[GitLab CI] D[test] E[security] F[build] G[deploy_staging] H[deploy_production] end I[(GitLab Registry)] J[Staging AWS EC2] K[Production Azure] A --> B --> C C --> D --> E --> F F --> I I --> J I --> K G --> J H --> K

Du code au registry, puis aux environnements staging (AWS) et production (Azure).

Le pipeline construit l’image et la pousse dans le GitLab Container Registry. Terraform (via les jobs de déploiement ou en manuel) crée ou met à jour l’infrastructure ; les instances EC2 et le service Azure tirent l’image depuis le registry et lancent le conteneur.

Figure 2 — Pipeline GitLab CI/CD (stages et flux)

flowchart TB subgraph pipeline[Pipeline] S1[test] S2[security] S3[build] S4[deploy_staging] S5[deploy_production] end Commit[Commit / Push] --> S1 --> S2 --> S3 S3 --> |image| REG[(Registry)] S3 --> S4 S4 --> |terraform apply| AWS[AWS EC2] S4 --> S5 S5 --> |manual gate| AZ[Azure App Service] REG --> AWS REG --> AZ

Enchaînement des stages : test, security, build, puis déploiement staging (automatique) et production (manuel).

Chaque stage s’exécute après le succès du précédent. Le stage build produit l’image et la pousse vers le registry. Les stages deploy_staging et deploy_production exécutent Terraform pour mettre à jour l’infrastructure et faire en sorte que l’instance EC2 ou Azure utilise la nouvelle image.

Staging vs production

Staging (AWS) : environnement de préproduction. On y déploie une version pour valider le comportement avant la mise en production. Ici : une instance EC2 qui exécute le conteneur Docker.

Production (Azure) : environnement final exposé aux utilisateurs. Ici : une Web App Azure (App Service for Containers) qui tire l’image depuis le GitLab Registry et expose l’application.

4. Développement de l’application Node.js (Partie A)

Objectif — Créer une application web minimale avec Node.js et Express : routes / et /health, configuration par variable d’environnement PORT, structure avec src/app.js (application Express) et src/server.js (démarrage du serveur).

Procédure

  • Créer le dossier du projet (ex. tp-devsecops-app), se placer dedans.
  • Initialiser le projet npm : npm init -y.
  • Installer Express : npm install express.
  • Créer le dossier src. Créer src/app.js : application Express avec les routes GET / et GET /health. Exporter l’objet app.
  • Créer src/server.js : importer app, lire process.env.PORT (défaut 3000), appeler app.listen(PORT, ...).
  • Dans package.json, définir "main": "src/server.js" et le script "start": "node src/server.js".
  • Créer .env.example avec PORT=3000. Créer .env (même contenu) pour le développement local. Ajouter à .gitignore : node_modules, .env, *.log.
  • Lancer npm start, ouvrir http://localhost:3000 et http://localhost:3000/health.

Fichiers complets

package.json (extrait essentiel) :

{
  "name": "tp-devsecops-app",
  "version": "1.0.0",
  "main": "src/server.js",
  "scripts": {
    "start": "node src/server.js",
    "test": "jest"
  },
  "dependencies": {
    "express": "^4.18.2"
  }
}

src/app.js :

const express = require('express');
const app = express();

app.get('/', (req, res) => {
  res.json({
    message: 'Bienvenue sur l\'application DevSecOps TP',
    version: '1.0'
  });
});

app.get('/health', (req, res) => {
  res.status(200).json({
    status: 'ok',
    timestamp: new Date().toISOString()
  });
});

module.exports = app;

src/server.js :

const app = require('./app');
const PORT = process.env.PORT || 3000;

app.listen(PORT, () => {
  console.log(`Serveur démarré sur http://localhost:${PORT}`);
});

.env.example :

PORT=3000

.gitignore :

node_modules
.env
*.log
.DS_Store
Pourquoi — process.env.PORT permet de configurer le port sans modifier le code (Docker, cloud). Le fichier .env n’est pas versionné pour éviter d’y mettre des secrets.

Résultat attendu

L’application répond en JSON sur / (message, version) et sur /health (status: "ok", timestamp).

Checkpoint A — npm start fonctionne ; les deux routes sont accessibles. Arrêter le serveur (Ctrl+C) avant de continuer.
Insérer capture : réponse / et /health (navigateur ou curl).

Erreurs fréquentes et correctifs

Port déjà utilisé : changer PORT dans .env ou arrêter le processus. Module introuvable : exécuter npm install depuis la racine du projet. Erreur « Cannot find module './app' » : vérifier que src/app.js existe et que vous lancez depuis la racine (node src/server.js).

5. Tests automatisés avec Jest et Supertest (Partie B)

Objectif — Mettre en place des tests avec Jest et Supertest pour les routes / et /health, exécutables en local et dans le pipeline CI.

Procédure

  • Installer Jest et Supertest en devDependencies : npm install --save-dev jest supertest.
  • Dans package.json, ajouter (ou compléter) le script "test": "jest". Optionnel : "jest": { "testMatch": ["**/__tests__/**/*.test.js"] } pour cibler les fichiers de test.
  • Créer le dossier __tests__ à la racine. Créer __tests__/app.test.js : importer supertest et app depuis ../src/app, écrire deux tests — GET / (status 200, body.message contient "DevSecOps") et GET /health (status 200, body.status === 'ok').
  • Lancer npm test.

Fichier de test complet

const request = require('supertest');
const app = require('../src/app');

describe('Routes', () => {
  test('GET / retourne un message', async () => {
    const res = await request(app).get('/');
    expect(res.status).toBe(200);
    expect(res.body.message).toContain('DevSecOps');
  });

  test('GET /health retourne status ok', async () => {
    const res = await request(app).get('/health');
    expect(res.status).toBe(200);
    expect(res.body.status).toBe('ok');
  });
});
Commande
npm test

Résultat attendu

Sortie du type « Tests: 2 passed ».

Checkpoint B — Tous les tests passent en local.
Insérer capture : sortie npm test.

Échec volontaire puis correction

Modifier temporairement src/app.js (par exemple changer le message ou le status) pour faire échouer un test. Relancer npm test : le test concerné doit être en échec. Restaurer le code correct et vérifier que les tests repassent.

Erreurs fréquentes et correctifs

Jest ne trouve pas les tests : placer les fichiers dans __tests__ avec l’extension .test.js ou configurer testMatch dans package.json. Timeout : augmenter avec jest.setTimeout(5000) si nécessaire.

6. Conteneurisation avec Docker (Partie C)

Objectif — Produire une image Docker de l’application avec Node 20, la construire et l’exécuter en local pour valider le comportement en conteneur (port 3000).

Procédure

  • Créer un Dockerfile à la racine du projet (voir ci-dessous).
  • Créer un .dockerignore pour exclure node_modules, .env, .git, *.log, __tests__ du contexte de build.
  • Builder l’image : docker build -t tp-devsecops-app:latest .
  • Lancer le conteneur : docker run -p 3000:3000 --name tp-app tp-devsecops-app:latest. Tester http://localhost:3000/health.
  • Consulter les conteneurs : docker ps. Consulter les logs : docker logs tp-app. Arrêter : docker stop tp-app ; supprimer : docker rm tp-app.

Dockerfile complet

# Image de base : Node.js 20 (LTS)
FROM node:20-alpine

# Répertoire de travail dans le conteneur
WORKDIR /app

# Copier les fichiers de dépendances
COPY package*.json ./

# Installer les dépendances (production uniquement)
RUN npm ci --only=production

# Copier le code source (y compris src/)
COPY . .

# Port exposé par l'application
EXPOSE 3000

# Variable d'environnement par défaut
ENV PORT=3000

# Point d'entrée : lancer le serveur Node
CMD ["node", "src/server.js"]

Explication ligne par ligne

  • FROM node:20-alpine — Image légère avec Node.js 20.
  • WORKDIR /app — Répertoire de travail dans le conteneur.
  • COPY package*.json ./ — Fichiers nécessaires pour npm ci.
  • RUN npm ci --only=production — Installation des dépendances sans devDependencies (Jest, etc.).
  • COPY . . — Copie du reste du code (le .dockerignore exclut ce qui ne doit pas entrer).
  • EXPOSE 3000 — Documente le port d’écoute.
  • ENV PORT=3000 — Valeur par défaut pour l’application.
  • CMD ["node", "src/server.js"] — Commande exécutée au démarrage du conteneur.

7. Pipeline GitLab CI/CD : qualité et sécurité (Partie D + E)

Objectif — Définir un pipeline GitLab avec les stages test, security, build, puis (section 8) publication dans le registry et (sections 10–11) déploiement. Faire échouer le pipeline en introduisant un faux secret, puis corriger.

Fichier .gitlab-ci.yml complet (commenté)

À la racine du projet, créer .gitlab-ci.yml. Ce fichier décrit les stages (étapes) et les jobs. Les variables CI_REGISTRY_* sont fournies automatiquement par GitLab pour s’authentifier au Container Registry.

# ========== Stages (ordre d'exécution) ==========
stages:
  - test
  - security
  - build
  - deploy_staging
  - deploy_production

variables:
  NODE_VERSION: "20"

# ========== Stage test : exécuter les tests Jest ==========
test:
  stage: test
  image: node:${NODE_VERSION}-alpine
  before_script:
    - npm ci
  script:
    - npm test

# ========== Stage security : détection de secret exposé ==========
# Si le code contient un motif type API_KEY=sk-demo-..., le job échoue
security:
  stage: security
  image: alpine:latest
  script:
    - |
      if grep -rE 'API_KEY\s*=\s*["\x27]?sk-[a-zA-Z0-9-]+' --include="*.js" --include="*.json" . 2>/dev/null; then
        echo "ERREUR : secret ou faux secret détecté (API_KEY=sk-...). Ne pas exposer de clés dans le code."
        exit 1
      fi
      echo "Aucun secret suspect détecté."

# ========== Stage build : construction de l'image Docker ==========
# Le login et le push vers le registry sont détaillés en section 8
build:
  stage: build
  image: docker:24
  services:
    - docker:24-dind
  variables:
    DOCKER_TLS_CERTDIR: "/certs"
  before_script:
    - docker info
  script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
    - docker build -t $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA -t $CI_REGISTRY_IMAGE:latest .
    - docker push $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA
    - docker push $CI_REGISTRY_IMAGE:latest

# ========== Deploy staging (AWS) : automatique sur branche develop ==========
deploy_staging:
  stage: deploy_staging
  image: hashicorp/terraform:latest
  script:
    - cd terraform/aws-staging
    - terraform init
    - terraform apply -auto-approve -var "docker_image=$CI_REGISTRY_IMAGE" -var "docker_tag=$CI_COMMIT_SHORT_SHA" -var "registry_url=$CI_REGISTRY" -var "registry_user=$CI_REGISTRY_USER" -var "registry_password=$CI_REGISTRY_PASSWORD"
  rules:
    - if: $CI_COMMIT_BRANCH == "develop"

# ========== Deploy production (Azure) : manuel (gate) sur branche main ==========
deploy_production:
  stage: deploy_production
  image: hashicorp/terraform:latest
  when: manual
  script:
    - cd terraform/azure-production
    - terraform init
    - terraform apply -auto-approve -var "docker_image=$CI_REGISTRY_IMAGE" -var "docker_image_tag=latest" -var "registry_url=https://$CI_REGISTRY" -var "registry_user=$CI_REGISTRY_USER" -var "registry_password=$CI_REGISTRY_PASSWORD"
  rules:
    - if: $CI_COMMIT_BRANCH == "main"

Démonstration pédagogique : faux secret

  1. Injecter un faux secret : dans un fichier (par ex. src/app.js ou un fichier de config factice), ajouter une ligne du type API_KEY=sk-demo-123456789 (uniquement des données fictives). Commiter et pousser.
  2. Résultat attendu : le job security du pipeline échoue (grep trouve le motif et le script fait exit 1).
  3. Corriger : supprimer la ligne contenant le faux secret. Remplacer par l’usage d’une variable d’environnement (ex. process.env.API_KEY) et documenter dans le README ou dans GitLab CI/CD Variables que les secrets ne doivent jamais être dans le code. Commiter et pousser.
  4. Résultat attendu : le pipeline repasse au vert (test, security, build).
N’utiliser que des faux secrets (ex. sk-demo-123456789) pour cet exercice. En production, les clés sont dans GitLab CI Variables (masked) ou dans un secret manager.

Lecture des logs GitLab

Dans GitLab : Build → Pipelines. Cliquer sur le pipeline concerné, puis sur chaque job pour afficher les logs. Les messages d’erreur (script security, échec des tests) s’affichent dans ces logs.

Résultat attendu

Pipeline vert après correction : stages test, security et build en succès.

Checkpoint D + E — Pipeline configuré ; démonstration échec (secret) puis succès (correction).
Insérer capture : pipeline en échec (job security) puis pipeline corrigé (tous les jobs verts).

Erreurs fréquentes et correctifs

Job build échoue (Docker) : vérifier que le runner a accès à Docker (service dind). Indentation YAML : utiliser des espaces, pas des tabulations. Job test échoue : vérifier que npm ci et npm test fonctionnent en local.

8. Publication de l’image Docker dans GitLab Container Registry

Objectif — Compléter le job build (ou créer un job dédié) pour : s’authentifier au GitLab Container Registry, construire l’image, la taguer avec le SHA du commit et latest, puis la pousser vers le registry.

Variables GitLab pour le Registry

GitLab fournit automatiquement dans chaque job CI :

  • CI_REGISTRY — URL du registry (ex. registry.gitlab.com).
  • CI_REGISTRY_IMAGE — URL complète de l’image du projet (ex. registry.gitlab.com/groupe/tp-devsecops-app).
  • CI_REGISTRY_USER et CI_REGISTRY_PASSWORD — Identifiants pour docker login.

Job build complet (login + build + tag + push)

build:
  stage: build
  image: docker:24
  services:
    - docker:24-dind
  variables:
    DOCKER_TLS_CERTDIR: "/certs"
  before_script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
  script:
    - docker build -t $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA -t $CI_REGISTRY_IMAGE:latest .
    - docker push $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA
    - docker push $CI_REGISTRY_IMAGE:latest

Tags : $CI_COMMIT_SHORT_SHA assure la traçabilité (chaque commit a une image taguée) ; latest pointe vers la dernière image buildée (pratique pour staging). En production on peut utiliser un tag de release (ex. v1.0.0) poussé manuellement ou par un job conditionné sur un tag Git.

Résultat attendu

Après un push, le pipeline exécute le job build ; l’image apparaît dans GitLab : Deploy → Container Registry. On y voit les tags (ex. abc1234, latest).

Checkpoint — Image visible dans le Container Registry du projet.
Insérer capture : pipeline (job build vert) et vue Container Registry avec les tags.

9. Infrastructure as Code avec Terraform (AWS staging / Azure production)

Terraform en quelques notions

Provider — Module qui communique avec un cloud (ex. hashicorp/aws, hashicorp/azurerm). Déclaré dans terraform { required_providers { ... } } et configuré via provider "aws" (région, credentials).

Resource — Élément d’infrastructure (ex. aws_instance, azurerm_linux_web_app). Chaque ressource a un type et un nom logique.

Variable — Paramètre d’entrée (fichier variables.tf ou .tfvars) pour rendre la config réutilisable (région, type d’instance, URL d’image, etc.).

Output — Valeur exportée après apply (IP publique, URL) pour exploitation manuelle ou par d’autres outils.

State — Fichier (local ou distant) qui enregistre l’état des ressources. Terraform l’utilise pour calculer les changements. Ne pas modifier à la main.

plan / apply / destroy — terraform plan affiche les changements sans les appliquer ; terraform apply crée ou modifie les ressources ; terraform destroy les supprime.

Figure 3 — Terraform multi-cloud (AWS staging / Azure production)

Un même outil Terraform décrit l’infrastructure des deux clouds.

Les dossiers terraform/aws-staging/ et terraform/azure-production/ contiennent chacun un provider et des ressources propres à chaque cloud. Les variables (image, tag, credentials registry) sont injectées pour déployer la même image Docker.

10. Déploiement en staging sur AWS (Partie F)

Objectif — Créer l’infrastructure de staging sur AWS avec Terraform : VPC par défaut (ou usage du default VPC), Security Group (HTTP 80, SSH 22), instance EC2. Le script user_data installe Docker, se connecte au GitLab Registry, tire l’image et lance le conteneur sur le port 80 (mapping 80→3000).

Structure des fichiers Terraform AWS

Dans terraform/aws-staging/ : main.tf (provider, security group, instance, user_data, clé SSH), variables.tf, outputs.tf, terraform.tfvars.example (sans secrets réels).

main.tf (complet)

terraform {
  required_providers {
    aws    = { source = "hashicorp/aws", version = "~> 5.0" }
    tls    = { source = "hashicorp/tls", version = "~> 4.0" }
  }
}

provider "aws" {
  region = var.aws_region
}

# Utiliser la VPC par défaut
data "aws_vpc" "default" {
  default = true
}
data "aws_subnets" "default" {
  filter { name = "vpc-id"; values = [data.aws_vpc.default.id] }
}

# Security Group : HTTP 80 (accès web) et SSH 22 (dépannage)
resource "aws_security_group" "staging_sg" {
  name        = "tp-devsecops-staging-sg"
  description = "HTTP 80, SSH 22 pour TP staging"
  vpc_id      = data.aws_vpc.default.id

  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

# Clé SSH pour se connecter à l'instance (optionnel)
resource "tls_private_key" "staging" {
  algorithm = "RSA"
  rsa_bits  = 2048
}
resource "aws_key_pair" "staging" {
  key_name   = "tp-staging-key"
  public_key = tls_private_key.staging.public_key_openssh
}

# AMI Amazon Linux 2
data "aws_ami" "amazon_linux_2" {
  most_recent = true
  owners      = ["amazon"]
  filter { name = "name"; values = ["amzn2-ami-hvm-*-x86_64-gp2"] }
}

# Script user_data : installer Docker, login registry, pull, run
resource "aws_instance" "staging" {
  ami                    = data.aws_ami.amazon_linux_2.id
  instance_type           = var.instance_type
  key_name               = aws_key_pair.staging.key_name
  vpc_security_group_ids = [aws_security_group.staging_sg.id]
  subnet_id              = tolist(data.aws_subnets.default.ids)[0]

  user_data = <<-EOT
#!/bin/bash
set -e
yum update -y
yum install -y docker
systemctl start docker
systemctl enable docker
echo "${var.registry_password}" | docker login -u "${var.registry_user}" --password-stdin ${var.registry_url}
docker pull ${var.docker_image}:${var.docker_tag}
docker stop app 2>/dev/null || true
docker rm app 2>/dev/null || true
docker run -d --name app --restart unless-stopped -p 80:3000 ${var.docker_image}:${var.docker_tag}
EOT

  tags = { Name = "tp-devsecops-staging" }
}

variables.tf

variable "aws_region" {
  type        = string
  default     = "eu-west-1"

12. Chaîne de livraison : de GitLab à AWS/Azure (Partie H)

Cette section détaille pas à pas comment l’image Docker est construite, stockée, puis récupérée par AWS et Azure, et ce qui est manuel ou automatisé.

Figure 4 — Flux Docker (build / tag / push / pull / run)

flowchart LR subgraph CI[GitLab CI] B[docker build] T[tag] P[push] end B --> T --> P P --> REG[(Registry)] REG --> PULL1[pull] REG --> PULL2[pull] PULL1 --> RUN1[run] PULL2 --> RUN2[run] RUN1 --> AWS[EC2] RUN2 --> AZ[Azure]

Construction et publication dans le registry, puis récupération par les environnements cloud.

A. Construction de l’image Docker

Qui construit l’image ? — Le runner GitLab CI (machine qui exécute les jobs du pipeline).

À quel moment ? — Lors du stage build, après les stages test et security.

À partir de quel fichier ? — Le Dockerfile à la racine du projet (et le contexte : code source, package.json, etc.).

Avec quelles commandes ? — docker build -t <image>:<tag> . Le point désigne le contexte (répertoire courant). Les tags utilisés sont en général $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA et $CI_REGISTRY_IMAGE:latest.

B. Stockage dans GitLab Container Registry

Où l’image est stockée ? — Dans le GitLab Container Registry du projet (Deploy → Container Registry). L’URL est du type registry.gitlab.com/<groupe>/<projet>.

Variables GitLab :

  • CI_REGISTRY — URL du registry (ex. registry.gitlab.com).
  • CI_REGISTRY_IMAGE — URL complète de l’image du projet.
  • CI_REGISTRY_USER / CI_REGISTRY_PASSWORD — Identifiants pour docker login dans le job.

Tags : :$CI_COMMIT_SHORT_SHA (traçabilité), :latest (dernier build), :v1.0.0 (release). On ne déploie pas directement le code source car le conteneur garantit un environnement reproductible (même OS, mêmes dépendances) et évite d’installer Node.js sur les serveurs ; l’image est l’unité de déploiement.

C. Récupération par AWS (staging)

Terraform crée l’instance EC2. Au premier démarrage, l’instance exécute le script user_data qui : (1) installe Docker, (2) se connecte au registry GitLab (docker login), (3) télécharge l’image (docker pull), (4) lance le conteneur (docker run -d -p 80:3000 ...). Les identifiants du registry sont fournis via les variables Terraform (registry_user, registry_password), elles-mêmes renseignées par un fichier terraform.tfvars non versionné ou par des variables GitLab CI (dans les jobs de déploiement) exportées en TF_VAR_*. En production réelle, on stockerait ces secrets dans un secret manager (AWS Secrets Manager, SSM Parameter Store) et on les récupérerait au moment du déploiement au lieu de les passer en clair dans Terraform.

D. Récupération par Azure (production)

Terraform crée le service App Service (Linux Web App). Azure tire l’image depuis le GitLab Registry. Azure a besoin de : l’URL du registry, le username et le password (Deploy Token), et l’image + tag. Ces paramètres sont configurés dans la ressource azurerm_linux_web_app (image, tag, et app_settings pour DOCKER_REGISTRY_SERVER_*). WEBSITES_PORT=3000 indique à Azure quel port le conteneur écoute pour router le trafic HTTP.

E. Automatisation depuis GitLab

Stage deploy_staging : exécution automatique (par ex. sur la branche develop) — le job lance terraform init / apply dans terraform/aws-staging/ en passant l’image et le tag (variables CI). Stage deploy_production : manual gate (when: manual) sur la branche main — un opérateur déclenche le job pour déployer en production. GitLab Environments (staging, production) permettent d’associer ces jobs à des environnements et d’afficher les URLs. Dans les jobs de déploiement, GitLab exporte les variables définies dans Settings → CI/CD → Variables ; le script exporte TF_VAR_docker_tag=$CI_COMMIT_SHORT_SHA (ou équivalent) puis exécute terraform apply -auto-approve.

Bonnes pratiques : secrets

Ne jamais commiter de secrets (mots de passe, tokens) dans le code. Utiliser GitLab CI/CD Variables (Settings → CI/CD → Variables) avec l’option « Masked » pour les jobs ; pour Terraform, utiliser des variables sensibles (sensitive = true) et des fichiers .tfvars non versionnés ou un backend distant pour le state.

17. Nettoyage des ressources (coûts)

Pour éviter des facturations après le TP, détruire toutes les ressources créées.

AWS

Dans terraform/aws-staging/ : exécuter terraform destroy, confirmer par yes. Vérifier dans la console AWS (EC2 → Instances, VPC → Security groups) que l’instance et le security group ont été supprimés.

Azure

Dans terraform/azure-production/ : exécuter terraform destroy, confirmer par yes. Vérifier dans le portail Azure (Resource groups, App Service) que le groupe de ressources et la Web App ont été supprimés.

Les ressources actives (VM, Web App) génèrent des coûts. En cas de doute, supprimer le resource group entier dans Azure ou terminer l’instance EC2 dans AWS depuis la console.
